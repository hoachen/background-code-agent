# Background Code Agent with Gemini CLI Integration

This project is a TypeScript and Node.js application that acts as a background code agent. It receives code modification tasks via a web interface, processes them using Google's Gemini CLI, and displays the progress and results in real-time on the website.

## Features

-   **Web Interface**: Simple frontend (`public/index.html`, `public/client.js`, `public/style.css`) to submit tasks and view their progress and results.
    -   Real-time updates for task status, logs, and generated diffs via polling.
-   **Task Submission API**: Endpoints in `src/main.ts` (Express server) to submit tasks and query their status.
    -   Accepts repository URL, branch, natural language task description, and optional setup scripts.
-   **Background Task Processing (`src/agent.ts`)**:
    -   Tasks are processed asynchronously.
    -   Each task runs in an isolated temporary workspace (`workspaces/<taskId>`).
-   **Code Modification with Gemini CLI**:
    -   Clones the specified Git repository into the workspace.
    -   Executes user-defined setup scripts (e.g., `npm install`).
    -   Invokes `@google/gemini-cli` with the task description to attempt code modifications.
    -   **Note**: The effectiveness of `gemini-cli` depends on its capabilities for non-interactive code modification and understanding the provided prompts.
-   **Diff Generation**: Uses `git diff HEAD` within the workspace to capture changes made by Gemini CLI.
-   **Result Tracking**: Stores and displays task ID, status (pending, in_progress, completed, failed), summary, execution logs, generated diff, and a (dummy) Pull Request URL.
-   **Error Handling**: Captures and logs errors from various stages like cloning, setup, Gemini CLI execution, and diff generation. These are displayed in the UI.

## Project Structure

-   `src/`: Contains the TypeScript source code.
    -   `types.ts`: Defines core interfaces (`TaskRequest`, `TaskResult`).
    -   `agent.ts`: Contains the `BackgroundCodeAgent` class with the core task processing logic (cloning, setup, Gemini CLI execution, diffing, workspace management).
    -   `main.ts`: Sets up an Express server, serves the frontend, and provides API endpoints for task management.
-   `public/`: Contains static frontend files.
    -   `index.html`: The main HTML page for the web interface.
    -   `style.css`: CSS for the frontend.
    -   `client.js`: JavaScript for frontend logic (task submission, polling API, rendering results).
-   `workspaces/`: Temporary directory created at runtime to store cloned repositories for each task. (This directory is added to `.gitignore`).
-   `dist/`: Contains the compiled JavaScript code (generated by `tsc`).
-   `package.json`: Project metadata and dependencies (including `express`, `@google/gemini-cli`).
-   `tsconfig.json`: TypeScript compiler configuration.
-   `.gitignore`: Specifies intentionally untracked files.

## Prerequisites

-   [Node.js](https://nodejs.org/) (version 16.x or later is recommended, check dependencies of `@google/gemini-cli` for specific requirements).
-   [npm](https://www.npmjs.com/) (usually comes with Node.js).
-   **Git**: Required for cloning repositories. Must be installed and available in the system's PATH.
-   **Gemini CLI (`@google/gemini-cli`)**:
    -   This is installed as a project dependency via `npm install`. The agent invokes it using `npx @google/gemini-cli`.
    -   **IMPORTANT**: `@google/gemini-cli` will likely require a `GEMINI_API_KEY` environment variable to be set in the environment where the agent is run. Refer to the official Gemini CLI documentation for authentication details. Without proper authentication, Gemini CLI calls are expected to fail. This application does not manage API keys itself.

## Installation

1.  Clone the repository:
    ```bash
    git clone <your_repository_url> background-code-agent
    cd background-code-agent
    ```
2.  Install the dependencies:
    ```bash
    npm install
    ```

## Running the Application

1.  **Build the TypeScript code** (compile TS to JS in `dist/`):
    ```bash
    npm run build
    ```
2.  **Start the server**:
    ```bash
    npm start
    ```
    This command typically executes `node dist/main.js`. The server will log its listening port (usually `http://localhost:3000`).

3.  **Open the web interface**: Navigate to `http://localhost:3000` (or the logged port) in your web browser.

## How it Works

1.  **Frontend Interaction**:
    -   The user submits a task (repository URL, branch, task description, optional setup scripts) via the web form on `index.html`.
    -   `client.js` sends this data as a JSON payload to the backend API endpoint `POST /api/tasks`.
    -   `client.js` periodically polls `GET /api/tasks` to fetch the status of all tasks and dynamically updates the task list in the UI.

2.  **Backend API (`src/main.ts`)**:
    -   An Express server listens for HTTP requests.
    -   `POST /api/tasks`: Receives new task requests. It calls `agent.submitTask()` from `BackgroundCodeAgent` to queue the task.
    -   `GET /api/tasks`: Returns a JSON object of all tasks and their current statuses.
    -   `GET /api/tasks/:taskId`: Returns the current status of a specific task.
    -   Serves static files from the `public/` directory (for the frontend).

3.  **Task Processing (`src/agent.ts` - `BackgroundCodeAgent`)**:
    -   When a task is submitted, `submitTask()`:
        -   Assigns a unique `taskId`.
        -   Creates a path for a temporary workspace: `workspaces/<taskId>`.
        -   Stores initial task data (status: 'pending').
        -   Asynchronously calls `processTask()`.
    -   The `processTask()` method executes the following steps:
        1.  **Workspace Setup**: Creates the unique temporary directory.
        2.  **Clone Repository**: Uses `git clone` to fetch the specified repository and branch into the workspace.
        3.  **Run Setup Scripts**: If provided, executes each setup script sequentially within the cloned repository's directory (e.g., `npm install`). Logs `stdout` and `stderr` from these scripts.
        4.  **Execute Gemini CLI**: Constructs a command like `npx @google/gemini-cli "<task_description>"` and runs it within the repository's directory.
            -   *Crucial Dependency*: This step's success heavily depends on `gemini-cli`'s non-interactive capabilities, its ability to understand the prompt and modify files directly, and proper authentication (e.g., via `GEMINI_API_KEY`).
            -   `stdout` and `stderr` from Gemini CLI are logged.
        5.  **Generate Diff**: After Gemini CLI execution, `git diff HEAD` is run in the workspace to capture any file changes. The diff output is stored.
        6.  **Simulate Tests**: (Current implementation) A placeholder that simulates test execution. It can be made to fail if the task description includes "fail test".
        7.  **Log & Update Status**: Throughout processing, detailed logs are collected. The task's status (`in_progress`, `completed`, `failed`), summary, diff, error messages, and a dummy PR URL are updated.
        8.  **Workspace Cleanup**: The temporary workspace directory (`workspaces/<taskId>`) and its contents are deleted, regardless of task success or failure.

## API Endpoints

-   `POST /api/tasks`: Submit a new task.
    -   **Request Body**: `application/json`
        ```json
        {
          "repoUrl": "string",
          "branch": "string",
          "taskDescription": "string",
          "setupScripts": ["string", "..."] // Optional
        }
        ```
    -   **Success Response (201)**: `application/json`
        ```json
        {
          "message": "Task submitted successfully",
          "taskId": "string"
        }
        ```
    -   **Error Response (400, 500)**: `application/json` with error details.
-   `GET /api/tasks`: Get all tasks and their current status.
    -   **Success Response (200)**: `application/json`
        ```json
        {
          "taskId1": { /* TaskResult object */ },
          "taskId2": { /* TaskResult object */ }
        }
        ```
-   `GET /api/tasks/:taskId`: Get the status of a specific task.
    -   **Success Response (200)**: `TaskResult` JSON object.
    -   **Error Response (404)**: If task not found.

## Important Considerations for Gemini CLI

-   **Authentication**: The environment running this application **must** be configured with any necessary authentication for `@google/gemini-cli` to function (e.g., a `GEMINI_API_KEY` environment variable). This application does not handle or store API keys.
-   **Non-Interactive Use**: The integration heavily relies on `@google/gemini-cli`'s ability to:
    1.  Run non-interactively.
    2.  Interpret the provided task description as a prompt to modify code.
    3.  Modify files directly in the working directory.
    If `gemini-cli` requires interactive input or behaves differently, the core code modification step will fail or hang.
-   **Error Output & Logging**: The agent captures `stdout` and `stderr` from Gemini CLI. These are crucial for diagnosing issues with its execution. They are available in the task logs on the web UI.
-   **Effectiveness**: The quality and relevance of code modifications are entirely dependent on the capabilities of the version of Gemini CLI being used and the clarity of the task description.
-   **Resource Usage**: Each task clones a repository. For large repositories or many tasks, this can consume significant disk space and network bandwidth. No specific resource limits are currently implemented.
